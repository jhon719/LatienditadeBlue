Para que el sistema sea desplegable tanto en local como en la nube (Vercel/Neon) sin errores, la clave es la **abstracción de la configuración**. El "agente" (ya sea tú mismo programando, o una IA asistiendo) debe tratar el entorno como una variable y no como algo fijo.

Aquí tienes la propuesta de estructura para la nota de despliegue, diseñada para evitar fallos de configuración.

---

# [[08.01-Estrategia-de-Despliegue-y-Configuracion-de-Entornos]]

## 1. Visión General: El Modelo `DotEnv`

La regla de oro para evitar fallos es el uso estricto de archivos de variables de entorno (`.env`). Nunca debe haber credenciales "hardcoded" en el código.

* **`local`**: Usa un archivo `.env.local` (ignorado por Git).
* **`staging/production`**: Usa las variables del dashboard de Vercel (Environment Variables).
* **Estandarización**: Todos los entornos deben compartir la misma estructura de variables, incluso si sus valores cambian.

## 2. Configuración de Base de Datos (Neon + Prisma)

Para que la transición entre local y nube sea transparente, utilizaremos **Prisma** como orquestador:

* **Conexión en Desarrollo:** `DATABASE_URL` apunta a una base de datos local (PostgreSQL en Docker o instalación local).
* **Conexión en Producción (Neon):** `DATABASE_URL` (Direct Connection) y `DATABASE_URL_NON_POOLING` (para migraciones).
* **Sincronización:** El agente debe verificar que el archivo `schema.prisma` utilice `env("DATABASE_URL")` para que no haya que modificar el código al desplegar.

## 3. Gestión de Despliegue en Vercel

Para evitar el "it works on my machine" (funciona en mi equipo pero no en el servidor):

* **Build Commands:** Definir en Vercel el comando: `npx prisma generate && next build`. Esto asegura que los tipos de la base de datos estén actualizados antes de construir la web.
* **Edge Middleware:** Si el agente planea usar funciones Edge, debe separar la lógica de base de datos de los componentes que se ejecutan en el cliente.
* **Runtime Requirements:** Especificar Node.js 20+ en el `package.json` para garantizar compatibilidad con librerías modernas.

## 4. Estructura de Validación (Pre-flight Check)

Antes de que el agente ejecute cualquier despliegue, debe pasar este checklist:

1. **Validación de Schema:** `npx prisma validate` (no debe haber errores de sintaxis).
2. **Verificación de Variables:** Script `check-env.js` que valide que todas las variables necesarias (API Keys, URLs) existan en el entorno actual.
3. **Logs:** Implementar un sistema de logging centralizado (como Vercel Logs o un servicio externo) para identificar errores de conexión a Neon en tiempo real.

### ¿Cómo organizar la Introducción al Despliegue para el Agente?

Cuando le pidas ayuda al agente para desplegar, usa siempre este "prompt de contexto":

> *"Actúa como un arquitecto DevOps. Vamos a desplegar este módulo [Nombre del Módulo]. El entorno es Next.js, la base de datos es Neon, y el despliegue es en Vercel. Asegúrate de que todas las dependencias usen variables de entorno, que el build incluya la generación de Prisma Client y que las migraciones de BD sean compatibles con la conexión transaccional de Neon."*

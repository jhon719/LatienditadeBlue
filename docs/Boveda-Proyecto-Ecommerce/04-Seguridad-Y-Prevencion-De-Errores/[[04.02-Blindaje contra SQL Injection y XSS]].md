# [[04.02-Blindaje contra SQL Injection y XSS]]

Esta nota define los protocolos de seguridad obligatorios para prevenir las vulnerabilidades más críticas en aplicaciones web modernas: la inyección de código SQL y el Cross-Site Scripting (XSS).

---

## 1. Blindaje contra SQL Injection (SQLi)

El SQL Injection ocurre cuando un atacante logra inyectar comandos SQL maliciosos a través de las entradas del usuario (inputs), manipulando la consulta que llega a la base de datos PostgreSQL.

### La Regla de Oro: Nunca concatenar strings para crear queries

El error fatal es hacer esto: `prisma.$queryRaw("SELECT * FROM users WHERE email = '" + userInput + "'")`. **Nunca** hagas esto.

### Nuestras Soluciones Técnicas:

* **Uso del Prisma ORM (Abstracción Segura):**
* Prisma utiliza **consultas parametrizadas** por defecto. Cuando usas `prisma.product.findMany({ where: { category: userInput } })`, Prisma no concatena el string; envía la consulta y los datos por separado a PostgreSQL, asegurando que el motor de base de datos trate la entrada del usuario estrictamente como *dato* y nunca como *código*.
* **Solución:** Prioriza siempre la API de alto nivel de Prisma (`findUnique`, `findMany`, `create`, `update`).


* **Seguridad en Raw Queries (Si es estrictamente necesario):**
* Si el ORM no alcanza y necesitas `prisma.$queryRaw`, utiliza siempre la sintaxis de *template tag* (Prisma lo maneja por ti):


```typescript
// ✅ SEGURO: Prisma escapa los valores automáticamente
const products = await prisma.$queryRaw`SELECT * FROM Product WHERE category = ${userInput}`;

// ❌ PELIGROSO: Nunca uses string concatenation aquí
// const products = await prisma.$queryRaw("SELECT * FROM Product WHERE category = '" + userInput + "'");

```

## 2. Blindaje contra Cross-Site Scripting (XSS)

El XSS ocurre cuando un atacante inyecta scripts (normalmente JavaScript) en tu web para ejecutarlos en el navegador de otros usuarios, usualmente para robar sesiones (cookies) o redirigir contenido.

### Nuestras Soluciones Técnicas:

* **React 19 (Auto-Escaping):**
* Por defecto, **React escapa automáticamente** cualquier valor que renderices en el JSX. Si un atacante intenta inyectar `<script>alert('xss')</script>` en un campo de nombre de usuario, React lo renderizará como texto plano y no como una etiqueta ejecutable.


* **El peligro de `dangerouslySetInnerHTML`:**
* Existe una función en React que "apaga" esta protección: `dangerouslySetInnerHTML`.
* **Regla:** Está prohibido usar `dangerouslySetInnerHTML` con contenido que provenga directamente del usuario (comentarios, nombres, descripciones de producto).
* **Si es inevitable (ej. renderizar HTML desde un CMS):** Debes usar una librería de desinfección como **DOMPurify** antes de renderizar.


```bash
npm install dompurify @types/dompurify

```


```typescript
import DOMPurify from 'dompurify';

function SanitizedHTML({ htmlContent }) {
  const clean = DOMPurify.sanitize(htmlContent);
  return <div dangerouslySetInnerHTML={{ __html: clean }} />;
}

```

## 3. Seguridad de Capa Adicional: Headers y CSP

Además del código, debemos instruir al navegador para que desconfíe de fuentes externas. Implementamos esto en `next.config.ts` o mediante un `middleware.ts`.

### Content Security Policy (CSP)

El CSP le dice al navegador: *"Solo ejecuta scripts que vengan de mis dominios de confianza, ignora cualquier script inyectado inline"*.

En tu archivo `middleware.ts`, puedes definir cabeceras de seguridad:

```typescript
import { NextResponse } from 'next/server';
import type { NextRequest } from 'next/server';

export function middleware(request: NextRequest) {
  const nonce = Buffer.from(crypto.randomUUID()).toString('base64');
  
  // CSP estricto
  const cspHeader = `
    default-src 'self';
    script-src 'self' 'nonce-${nonce}' 'strict-dynamic';
    style-src 'self' 'unsafe-inline';
    img-src 'self' blob: data: https://res.cloudinary.com;
    connect-src 'self';
    frame-ancestors 'none';
  `.replace(/\s{2,}/g, ' ').trim();

  const requestHeaders = new Headers(request.headers);
  requestHeaders.set('x-nonce', nonce);
  requestHeaders.set('Content-Security-Policy', cspHeader);

  return NextResponse.next({
    request: { headers: requestHeaders },
  });
}

```

> **Nota:** Esto bloquea que alguien inyecte un `<script src="[atacante.com/mal.js](https://atacante.com/mal.js)">` en tu sitio, ya que el navegador solo ejecutará scripts que tú hayas autorizado explícitamente.

## 4. Checklist de Seguridad para el Equipo

Para mantener este blindaje, aplicamos estos puntos de control en cada revisión de código (*Code Review*):

* [ ] **No renderizado directo:** ¿Estamos usando `dangerouslySetInnerHTML`? Si sí, ¿está pasando por DOMPurify?
* [ ] **Validación en Servidor:** ¿Estamos confiando en los datos que llegan del frontend? (Todo input del usuario debe validarse en el *Server Action* usando `zod`, nunca confíes en la validación solo del lado del cliente).
* [ ] **Dependencias:** ¿Tenemos librerías desactualizadas? (Correr `npm audit` periódicamente para detectar vulnerabilidades en paquetes de terceros).
* [ ] **Cookies Seguras:** Como usamos **NextAuth v5**, asegúrate de que en la configuración de producción las cookies tengan `httpOnly: true`, `secure: true`, y `sameSite: 'lax'`. (Esto impide que scripts XSS roben la sesión del usuario).

# [[04.01-Idempotencia y Prevención de Doble Clic en APIs]]

Esta nota documenta las medidas de seguridad y robustez implementadas en `ecommerce-basictech` para evitar la creación de órdenes duplicadas, cargos indebidos o inconsistencias en la base de datos cuando un usuario, por impaciencia o error de red, presiona botones de acción múltiples veces.

---

## 1. Estrategia de Dos Capas

Para garantizar la integridad del sistema, aplicamos una defensa en profundidad:

1. **Capa Frontend (UX):** Bloqueo visual e inmediato del estado de carga usando React 19.
2. **Capa Backend (Database):** Bloqueo a nivel de base de datos usando una **Idempotency Key** con restricción de unicidad en PostgreSQL.

---

## 2. Capa Frontend: React 19 `useFormStatus`

Con la adopción de React 19 en tu proyecto, eliminamos la necesidad de estados complejos para cada botón. Usamos el hook `useFormStatus` para detectar automáticamente si el formulario padre se está enviando y deshabilitar el botón de forma nativa.

```tsx
"use client";

import { useFormStatus } from "react-dom";

export function SubmitButton({ label }: { label: string }) {
  const { pending } = useFormStatus();

  return (
    <button
      type="submit"
      disabled={pending} // Se bloquea automáticamente al enviarse
      className="w-full py-3 bg-azul-oscuro text-white disabled:bg-gray-400"
    >
      {pending ? "Procesando..." : label}
    </button>
  );
}

```

## 3. Capa Backend: Idempotencia con Prisma 7

La UI puede fallar si el usuario recarga la página o si el navegador hace un *retry* automático. Para evitar esto, obligamos al cliente a generar una **Idempotency Key** (un UUID único por intención de compra).

### A. Modificación del Schema (Prisma)

Añadimos un campo único en el modelo de `Order`. PostgreSQL rechazará cualquier intento de inserción con una llave duplicada, protegiendo contra *race conditions*.

```prisma
model Order {
  id              String   @id @default(uuid())
  // ... resto de campos
  
  // Campo de seguridad para prevenir duplicados
  idempotencyKey  String   @unique 
  
  createdAt       DateTime @default(now())
}

```

### B. Implementación en Server Action

En tu servidor, el flujo para crear una orden ahora verifica este campo:

```typescript
"use server";

import { prisma } from "@/lib/prisma";

export async function createOrderAction(formData: FormData, idempotencyKey: string) {
  try {
    // Intentamos crear la orden. Si ya existe la key, Prisma lanzará error.
    const newOrder = await prisma.order.create({
      data: {
        idempotencyKey,
        // ... resto de datos
      }
    });
    
    return { success: true, order: newOrder };
    
  } catch (error: any) {
    // Si el error es por violación de unicidad (P2002)
    if (error.code === 'P2002') {
      return { success: false, message: "Tu pedido ya está siendo procesado." };
    }
    throw error;
  }
}

```

---

## 4. Flujo de Generación de la Key

Para que esto sea efectivo, el componente de Checkout debe generar la `idempotencyKey` **antes** de montar el formulario:

1. El usuario entra al Checkout.
2. El componente genera un `crypto.randomUUID()` y lo guarda en un estado local o un campo oculto del formulario (`hidden input`).
3. Al presionar "Confirmar Pedido", se envía este UUID al servidor.
4. Si el usuario hace doble clic, el segundo intento envía el mismo UUID; PostgreSQL detecta que la `idempotencyKey` ya existe y aborta la transacción, devolviendo el mensaje de error "Tu pedido ya está siendo procesado" en lugar de crear una segunda orden.

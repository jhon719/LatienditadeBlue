# [[02.03-Sistema de Perfiles y Datos Privados de Admin]]

Esta nota especifica la arquitectura de seguridad, la gestión de perfiles de usuario, el Control de Acceso Basado en Roles (RBAC) y la protección estricta de datos personales sensibles dentro de **La Tiendita de Blue**.

---

## 1. Filosofía de Privacidad (Zero-Leak Policy)

Para garantizar la confianza de los coleccionistas y cumplir con las normativas de protección de datos, el sistema implementa una separación absoluta entre la **identidad pública** del usuario (comunidad, reseñas) y su **identidad privada de facturación/envío** (solo accesible por el propio usuario y el administrador).

### Matriz de Visibilidad de Datos

| Campo en DB | Tipo de Dato | ¿Visible Públicamente? | ¿Visible por el Admin? | Propósito |
| --- | --- | --- | --- | --- |
| `username` | Público | **Sí** | Sí | Identificador en reseñas y chat. |
| `avatarUrl` | Público | **Sí** | Sí | Imagen de perfil de Cloudinary. |
| `realName` | Sensible | **No** | Sí | Despacho de paquetes y boletas. |
| `dni` | Sensible | **No** | Sí | Obligatorio para facturación y aduanas. |
| `phoneNumber` | Sensible | **No** | Sí | Coordinación de entregas vía WhatsApp. |
| `address` | Sensible | **No** | Sí | Dirección física de envío de la figura. |

---

## 2. Definición del Modelo de Datos (Prisma SQL)

El esquema de la base de datos en PostgreSQL (vía Neon) estructura al usuario protegiendo los campos críticos y asignando un rol de manera nativa.

```prisma
enum Role {
  USER
  ADMIN
}

model User {
  id            String    @id @default(uuid())
  email         String    @unique
  role          Role      @default(USER)
  
  // Perfil Público
  username      String    @unique
  avatarUrl     String?   // URL optimizada de Cloudinary
  
  // Datos Privados y Sensibles
  realName      String?
  dni           String?   @unique // DNI de 8 dígitos para envíos en Perú
  phoneNumber   String?
  address       String?
  
  // Relaciones del Sistema
  orders        Order[]
  reviews       Review[]
  
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
}

```

## 3. Seguridad a Nivel de API (Next.js Server Actions)

Para evitar que un usuario malintencionado intente consultar datos de otro mediante herramientas de inspección de red, las peticiones al backend filtran estrictamente los campos devueltos en las consultas SQL.

### Acción Segura para Obtener Perfiles Públicos

Al renderizar el perfil de un usuario que dejó una reseña, se utiliza una proyección de Prisma que **excluye** explícitamente los campos sensibles:

```typescript
// src/app/actions/get-public-profile.ts
"use server";

import { prisma } from "@/lib/prisma";

export async function getPublicProfile(username: string) {
  return await prisma.user.findUnique({
    where: { username },
    select: {
      username: true,
      avatarUrl: true,
      role: true,
      // Los campos realName, dni, address y phoneNumber quedan completamente excluidos de la consulta
    }
  });
}

```

## 4. Control de Acceso de Administrador (RBAC Middleware)

Las vistas de gestión del inventario, control de preventas, visualización de vouchers de pago y despachos se agrupan bajo la ruta protegida `/admin/*`. El acceso se valida a nivel de infraestructura utilizando el middleware de Next.js.

```typescript
// src/middleware.ts
import { NextResponse } from "next/server";
import type { NextRequest } from "next/server";
import { getToken } from "next-auth/jwt";

export async function middleware(req: NextRequest) {
  const token = await getToken({ req });
  const isAuth = !!token;
  const isAdminRoute = req.nextUrl.pathname.startsWith("/admin");

  if (isAdminRoute) {
    if (!isAuth || token.role !== "ADMIN") {
      // Redirección sutil al Home si no cuenta con permisos de administrador
      return NextResponse.redirect(new URL("/", req.url));
    }
  }

  return NextResponse.next();
}

export const config = {
  matcher: ["/admin/:path*", "/profile/:path*"],
};

```

## 5. Diseño de Interfaces Protegidas (UI/UX)

La experiencia visual se adapta dinámicamente según la sesión activa, manteniendo la consistencia de marca (colores pasteles y estética amigable).

### Vista de Perfil del Cliente (`/profile`)

* **Privacidad Visual:** Los campos como el DNI o la dirección se muestran parcialmente enmascarados por defecto (ej: `45***128` o `Av. Larco ***, Miraflores`) junto a un botón de "Editar" para actualizarlos.
* **Control de Pedidos:** Muestra el historial de compras con el estado de sus preventas y el código de proceso único (`processCode`) necesario para los retiros en tienda.

### Panel de Control del Administrador (`/admin/dashboard`)

* **Visualización de Clientes:** Una tabla limpia que permite filtrar por DNI o número de pedido.
* **Gestión de Despachos:** El administrador puede ver directamente el nombre real del cliente, su teléfono de contacto rápido y su dirección de entrega para generar las etiquetas de envío de forma masiva sin pasos intermedios.
* **Estilo Visual:** El panel de administración conserva la tipografía de la marca (**Bebas Neue** en los encabezados) pero utiliza una distribución de información mucho más estructurada, limpia y orientada a la productividad.

---

> 🛡️ **Nota de Seguridad:** Todas las contraseñas se gestionan de manera externa mediante proveedores de autenticación seguros (NextAuth con Google/Credentials) cifrando las credenciales locales mediante `bcrypt` antes de impactar en la base de datos de Neon.

De hecho, para tiendas con una comunidad tan cercana como la de **La Tiendita de Blue** (donde muchos clientes ya te escriben directamente por WhatsApp para coordinar compras), el flujo de **"Cambio por Solicitud de Soporte"** es una salvación de UX.

Muchos usuarios se confunden con los correos de recuperación automática (que a veces caen a Spam o nunca los abren). Darle al Administrador el poder de generar una **clave temporal** y forzar al usuario a cambiarla al iniciar sesión es una solución robusta y súper humana.

Para documentar esto de forma impecable, crearemos una subnota especializada dentro de tu carpeta de seguridad: **`[[02.03.01-Autenticacion-Registro-y-Recuperacion]]`**.

# [[02.03.01-Autenticacion-Registro-y-Recuperacion]]

Esta nota detalla el funcionamiento técnico del motor de autenticación doble (Google + Credenciales), el flujo de registro y el sistema híbrido de recuperación de contraseñas (Autoservicio + Soporte de Administrador con cambio forzado).

---

## 1. Actualización del Modelo de Base de Datos (Prisma)

Para soportar el inicio de sesión tradicional y el flujo de cambio forzado por administración, extendemos el modelo `User` en tu base de datos PostgreSQL:

```prisma
// Adiciones al modelo User en [[01.03-Modelo de Base de Datos Prisma SQL]]
model User {
  id                 String    @id @default(uuid())
  email              String    @unique
  username           String    @unique
  role               Role      @default(USER)
  
  // Autenticación con Credenciales
  passwordHash       String?   // Opcional, ya que los usuarios de Google no tienen contraseña local
  
  // Flujo de Recuperación por Soporte (Admin)
  mustChangePassword Boolean   @default(false) // Si es true, se bloquea la navegación hasta que la cambie
  
  // Relación con cuentas OAuth (Google, etc.)
  accounts           Account[] 
  
  createdAt          DateTime  @default(now())
  updatedAt          DateTime  @updatedAt
}

model Account {
  id                String  @id @default(uuid())
  userId            String
  type              String
  provider          String
  providerAccountId String
  user              User    @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@unique([provider, providerAccountId])
}

```

## 2. Flujo de Login y Registro Doble

El sistema utiliza **NextAuth.js (Auth.js)** para gestionar de forma segura los dos métodos de entrada:

### Opción A: Registro e Inicio con Google (OAuth 2.0)

* **Registro:** El usuario hace clic en "Registrarse con Google". NextAuth extrae el correo, genera un `username` sugerido basado en su nombre (ej. `pedro.gomez12`), descarga su foto de Google (`avatarUrl`) para guardarla en la sesión y deja el campo `passwordHash` como `null`.
* **Seguridad:** Si el usuario intenta registrarse después usando su correo de Google en el formulario de credenciales, el sistema le advertirá amigablemente: *"Esta cuenta ya está vinculada a Google. Por favor, inicia sesión con tu cuenta de Google."*

### Opción B: Registro Clásico con Credenciales

* El formulario solicita únicamente `username`, `email` y `password` (minimizando la fricción inicial).
* La contraseña se procesa y encripta con **`bcryptjs`** con un factor de coste (*salt rounds*) de 10 antes de guardarse en la base de datos como `passwordHash`.

## 3. Flujo de "Cambio por Solicitud" (Soporte Admin)

Este es el proceso diseñado para resolver problemas de acceso de forma directa a través del Administrador:

```text
[Cliente sin Acceso] ──(Pide ayuda vía WhatsApp)──> [Administrador]
                                                          │
                                            (Busca usuario en /admin)
                                                          │
                                           (Genera Clave Temporal)
                                                          │
                                       • passwordHash = Bcrypt(temporal)
                                       • mustChangePassword = true
                                                          │
[Cliente] <───(Recibe "Blue-Temp-123" por WhatsApp)───────┘
   │
 (Inicia sesión con su email y clave temporal)
   │
 (SISTEMA INTERCEPTA: mustChangePassword === true)
   │
 └───> [Vista Forzada de Cambio de Clave] ──(Guarda nueva clave)──> [Sesión Activa]

```

### Paso 1: Acción del Administrador (Server Action)

El administrador busca al usuario en el panel de control y genera una contraseña temporal (ej: `Blue-Temp-8391`). Al hacer esto, se dispara esta acción:

```typescript
// src/app/actions/admin/reset-user-password.ts
"use server";

import { prisma } from "@/lib/prisma";
import bcrypt from "bcryptjs";

export async function adminResetPassword(userId: string, tempPasswordHex: string) {
  // Asegurar que el que ejecuta la acción es realmente un ADMIN (RBAC)
  // ... validación de sesión de administrador ...

  const hashedTempPassword = await bcrypt.hash(tempPasswordHex, 10);

  await prisma.user.update({
    where: { id: userId },
    data: {
      passwordHash: hashedTempPassword,
      mustChangePassword: true, // Activamos la bandera de bloqueo
    },
  });

  return { success: true, message: "Contraseña temporal asignada con éxito" };
}

```

### Paso 2: Intercepción del Middleware al Iniciar Sesión

Cuando el cliente inicia sesión con la contraseña temporal, NextAuth carga la sesión. Nuestro middleware o los callbacks de NextAuth detectan la propiedad `mustChangePassword: true` y limitan su acceso en toda la plataforma:

```typescript
// En NextAuth callbacks (session)
callbacks: {
  async jwt({ token, user }) {
    if (user) {
      token.role = user.role;
      token.mustChangePassword = user.mustChangePassword;
    }
    return token;
  },
  async session({ session, token }) {
    if (session.user) {
      session.user.role = token.role;
      session.user.mustChangePassword = token.mustChangePassword;
    }
    return session;
  }
}

```

* **Comportamiento en la Web:** Si `session.user.mustChangePassword` es `true`, el Middleware de Next.js redirige automáticamente cualquier intento de navegación (excepto cerrar sesión) hacia la página `/auth/forced-reset`.

## 4. Interfaz de Cambio Forzado (`/auth/forced-reset`)

Para mantener el tono dulce, cercano y confiable de la marca, esta pantalla no debe parecer un error del sistema o un castigo.

* **Texto de Bienvenida (Mascota Bluet):** *"¡Hola coleccionista! Para mantener tu cuenta ultra segura tras tu solicitud de soporte, por favor ingresa tu nueva contraseña personalizada."*
* **Formulario:**
* Solicita: `Contraseña Nueva` y `Confirmar Contraseña`.
* **Validaciones en tiempo real:** Mínimo 8 caracteres, al menos un número o símbolo especial.
* Una vez que el usuario ingresa la nueva clave y pulsa "Guardar y Continuar", se ejecuta una acción que actualiza el `passwordHash` y restablece `mustChangePassword` a `false`. El cliente es redirigido con éxito a la tienda ya logueado con su nueva contraseña.

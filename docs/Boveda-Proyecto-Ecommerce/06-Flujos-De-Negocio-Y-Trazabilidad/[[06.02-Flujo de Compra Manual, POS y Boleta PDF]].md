Este módulo es fundamental para atender a aquellos clientes que prefieren canales de comunicación directa (WhatsApp/Instagram) o que simplemente tienen dudas antes de comprar. Aquí documentamos el flujo donde el administrador actúa como facilitador del proceso de venta.

---

# [[06.02-Flujo de Compra Manual, POS y Boleta PDF]]

## 1. Visión General: El Vendedor como Facilitador

Este flujo ocurre fuera de la pasarela automática. Es un proceso **semi-asistido**, donde el administrador genera el pedido manualmente tras haber cerrado la venta por chat o teléfono. La clave aquí es la **emisión inmediata del comprobante** y la reserva del inventario.

---

## 2. El Flujo de Trabajo (Work-flow)

### Paso 1: Registro Manual (Pedido "Back-office")

Cuando un cliente confirma la compra por chat, el administrador accede a la sección **"Crear Pedido"** en el Panel Administrativo (Módulo 05.03).

* **Selección de Productos:** Se buscan las figuras en el catálogo. El sistema valida stock en tiempo real y lo bloquea inmediatamente para este pedido.
* **Datos del Cliente:** Si el cliente ya existe en el CRM, se autocompletan sus datos. Si es nuevo, se crea su ficha en segundos.
* **Cálculo de Envío:** Se añade el costo de envío (Shalom/Olva) de forma manual o seleccionando una tarifa preestablecida por zona geográfica.

### Paso 2: Generación del Resumen de Pago (Proforma)

Antes de cobrar, el sistema genera un **PDF de Proforma** (enlace único) que el administrador envía al WhatsApp del cliente.

* Contiene: Detalle de la compra, montos, cuenta bancaria (Yape/Plin/Interbank) y QR personalizado.
* *Nota:* Este PDF sirve como recordatorio formal para que el cliente realice la transferencia.

### Paso 3: Validación del Comprobante (Re-vinculación con 05.02)

Una vez que el cliente transfiere, sube la captura de pantalla al enlace que recibió (o la envía por WhatsApp y el admin la carga al panel).

* El pedido entra automáticamente a la **Bandeja POS (05.02)** con estado `PENDING_VERIFICATION`.
* El administrador valida el voucher, presiona **"Aprobar"** y el flujo continúa.

### Paso 4: Emisión de Boleta Electrónica (Automatizada)

En el momento que se aprueba el pago:

* El sistema genera un **PDF de Boleta/Factura** (conectado a una API de facturación electrónica como *Nubefact, Izipay Facturación u otra*).
* El sistema envía automáticamente el PDF de la boleta y el número de pedido al correo electrónico del cliente.
* **Almacenamiento:** El PDF queda guardado en la "Ficha 360° del Cliente" en el módulo CRM para futuras consultas de garantía.

---

## 3. Arquitectura del Sistema de Facturación

Para que este flujo sea profesional y cumpla con las normativas locales (SUNAT), la arquitectura interna debe ser sólida:

* **Integración API (Facturador):** Tu panel administrativo no "crea" la boleta desde cero; el sistema envía los datos (RUC/DNI, Razón Social, Ítems, Precios) vía JSON a un proveedor de facturación electrónica.
* **Webhook de Respuesta:** El proveedor de facturación devuelve un link de descarga (`.pdf` y `.xml`) que tu sistema almacena en la base de datos de PostgreSQL vinculada al `order_id`.
* **Logs de Error:** Si la SUNAT rechaza la boleta (por un error en el DNI, por ejemplo), el sistema debe mostrar una alerta en el Dashboard Principal para que el administrador pueda corregir y reintentar con un solo clic.

---

## 4. Diferencias entre Flujo Automático vs. Manual

| Característica | Flujo Automático (06.01) | Flujo Manual (06.02) |
| --- | --- | --- |
| **Intervención Admin** | Nula | Alta |
| **Validación de Pago** | Instantánea (API) | Visual (Admin) |
| **Emisión de Boleta** | Automática (Post-pago) | Automática (Post-aprobación) |
| **Ideal para...** | Usuarios web, compras rápidas | Ventas por WhatsApp, Preventas |

---

## 5. Mejores Prácticas para el Admin

* **Plantillas de Mensajes:** Para el flujo manual, el panel debe tener botones de "Copiar Mensaje" (ej. *"Hola, aquí tienes el resumen de tu pedido, el monto total es S/ XX, cuenta bancaria Y..."*), para no tener que redactar lo mismo cada vez.
* **Historial de Cambios (Audit Log):** Como es un flujo manual, el sistema debe registrar quién aprobó qué pedido y cuándo, para evitar confusiones en el manejo del dinero.

### 1. El Flujo de "Enlace Copiable" (Manual-Assisted)

En lugar de que el sistema fuerce el envío, el proceso cambia a este modelo:

1. **Creación en el Panel:** Creas el pedido en el panel (como ya definimos).
2. **Generación del Link:** El sistema te muestra un botón grande: **"Copiar Enlace de Pago"**.
3. **Gestión por WhatsApp:**
* Tú vas a tu WhatsApp (Web o Desktop).
* Pegas el enlace junto con tu saludo personalizado.
* *Ventaja:* Tienes control total sobre el tono de la conversación. No parece un mensaje robotizado, sino una atención personalizada.


4. **Experiencia del Cliente:** Cuando el cliente hace clic en el enlace, **no necesita registro previo ni login complejo**. Llega a una página web (tu dominio) donde ve su resumen, tus cuentas y el botón para subir su comprobante.

### 2. Diseño de la "Página de Pago" (El link que copias)

Esta página debe estar diseñada para que el cliente no tenga dudas:

* **Cabecera:** Tu logo y un mensaje: *"Hola [Nombre Cliente], este es tu resumen de pedido"*.
* **Cuerpo Central (Tabla Resumen):** Los productos, cantidades y el costo de envío.
* **Sección de Pago:**
* Muestra tus cuentas (Yape, Plin, Interbank) en tarjetas claras.
* Incluye un **botón de carga de archivo** (input de tipo file) muy visible.


* **Pie de página:** Un botón de "Ya envié mi comprobante" que bloquea la subida una vez que se envía, para evitar archivos duplicados.

### 3. El Rol del Admin en este flujo

Tu rol es el de **"Orquestador"**:

* Tú envías el link.
* El cliente hace el trabajo sucio (subir el voucher).
* El sistema te avisa en la **[[05.02-Bandeja POS]]** apenas el cliente sube el archivo.
* Tú solo entras, miras la imagen y apruebas.

## 3. Diseño del Comprobante (Boleta Electrónica PDF)

El diseño debe ser profesional, minimalista y, sobre todo, **seguro**. Se utilizará una estructura de tabla clara para los ítems y elementos de seguridad para prevenir falsificaciones.

### A. Elementos Visuales y Estructurales

* **Encabezado:** Logo de "La Tiendita de Blue" en alta resolución, datos de la empresa (RUC, Dirección Fiscal, Teléfono, Web).
* **Datos del Receptor:** Nombre/Razón Social, DNI/RUC, Dirección de envío, Fecha y Hora exacta de emisión (formato `DD/MM/YYYY HH:mm:ss`).
* **Tabla de Detalles:** Columnas alineadas: `Descripción`, `Cantidad`, `Precio Unitario`, `Total`.
* **Totales:** Subtotal, IGV (según normativa local) y Monto Total.

### B. Elementos Antifalsificación (Security Layer)

Para que el cliente y cualquier autoridad puedan verificar la autenticidad del documento:

1. **Código QR Dinámico:** Ubicado en el pie de página. Al escanearlo, debe redirigir a una página de tu sistema (o del facturador electrónico) que confirme: *"Este documento es auténtico y fue emitido el [Fecha/Hora]"*.
2. **Hash de Seguridad (Código Alfanumérico):** Un código único de 16 caracteres impreso en la parte inferior, generado mediante una función criptográfica (`SHA-256`) que combina el `OrderID` + `RUC` + `FechaHora`. Esto hace que el PDF sea prácticamente imposible de replicar manualmente.
3. **Marca de Agua:** Un logo sutil con baja opacidad en el centro del PDF.
4. **Sello Digital:** Una firma digital que bloquea el PDF contra ediciones posteriores (imposible modificar montos o nombres usando editores básicos).
